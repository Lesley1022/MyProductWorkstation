# Skill: design_data_tracking

## 所属Agent
数据埋点Agent (Data Agent)

## 描述
生成数据埋点文档，明确需要采集的事件、事件属性、上报时机、关联功能ID等。用于指导开发和数据团队实施埋点。同时需考虑通用规则中操作日志的要求，并对明文查看等特殊操作进行埋点。

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| `project_id` | string | 是 | 项目标识 |
| `platform_id` | string | 是 | 平台标识 |
| `version` | string | 是 | 版本号 |
| `common_rules` | object | 是 | DAO OS通用规则文档内容 |
| `previous_docs` | array | 是 | 已生成的文档列表，必须包含PRD、功能列表、HTML交互说明 |
| `pending_confirms` | array | 是 | 待确认项列表 |

## 输出参数
| 参数名 | 类型 | 描述 |
|--------|------|------|
| `document_content` | string | 按照“数据埋点模板”生成的完整文档内容 |

## 内部逻辑
1. 加载“数据埋点模板”。
2. 从PRD、功能列表和交互说明中识别所有需要埋点的用户行为，例如：
   - 页面浏览（PV）
   - 按钮点击
   - 表单提交
   - 搜索
   - 错误提示
   - 明文查看操作（需单独记录）
3. 为每个事件生成唯一事件ID，格式如 `evt_page_view_dashboard`。
4. 定义每个事件的属性：
   - 公共属性：用户ID、时间戳、页面URL、操作人、操作时间、操作内容等（与规则4-操作日志要求一致）
   - 特有属性：如按钮名称、输入内容、错误码等
5. 明确上报时机（实时、延迟批量）。
6. 将事件与功能ID关联，便于分析功能使用情况。
7. 确保埋点上报的数据若包含敏感字段，需遵循脱敏规则（但通常操作日志已脱敏）。
8. 生成埋点文档，包含事件列表表格。
9. 返回文档内容。

## 注意事项
- 埋点设计应考虑数据分析需求，避免采集无用数据，同时不遗漏关键指标。
- 属性命名应规范，便于后续处理。
- 操作日志相关事件需覆盖规则4要求的全部内容。
