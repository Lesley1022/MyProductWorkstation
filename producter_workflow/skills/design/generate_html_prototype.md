# Skill: generate_html_prototype

## 所属Agent
设计Agent (Design Agent)

## 描述
生成高保真HTML原型，基于Tailwind CSS框架，包含动态数据模拟和微交互效果。原型支持多界面导航，并可在浏览器中演示。生成的原型必须严格遵循DAO OS通用规则。

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| `project_id` | string | 是 | 项目标识 |
| `platform_id` | string | 是 | 平台标识 |
| `version` | string | 是 | 版本号 |
| `common_rules` | object | 是 | DAO OS通用规则文档内容，必须传入以确保原型符合规范 |
| `previous_docs` | array | 是 | 已生成的文档列表，至少包含信息架构/功能列表、业务流程图 |
| `modification_feedback` | object | 否 | 若是在界面审批循环中传入的用户修改意见，包含 `{界面ID: 意见列表}` 的结构 |
| `pending_confirms` | array | 是 | 待确认项列表 |

## 输出参数
| 参数名 | 类型 | 描述 |
|--------|------|------|
| `html_content` | string | 完整的HTML代码，包含CSS和JavaScript，可独立运行 |
| `screenshots` | array | 界面缩略图列表（Base64编码或URL），每个元素对应一个界面 |
| `界面列表` | array | 每个界面的元数据，包含 `id`（界面ID）、`title`（界面标题） |

## 内部逻辑
1. 解析信息架构/功能列表，确定需要设计的界面清单（如登录页、仪表盘、资源列表页、资源详情页、交易订单页等）。每个界面分配唯一ID（如`page-dashboard`）。
2. 若提供了 `modification_feedback`，则优先根据反馈调整对应界面的设计。
3. 对于每个界面：
   - 基于功能点描述，设计布局和组件。
   - 使用Tailwind CSS编写HTML结构，确保响应式。
   - 添加必要的JavaScript模拟动态数据（如使用Chart.js展示负荷曲线，使用定时器模拟实时数据变化）。
   - 实现界面间的跳转（通过点击按钮切换显示/隐藏或改变hash）。
   - 添加微交互：按钮悬停效果、输入框聚焦样式、表单验证提示、加载动画等。
   - **遵循DAO OS通用规则**：
     - 查询条件：文本框设置`maxlength=50`，日期选择器格式为`YYYY-MM-DD`，默认值、前后切换等。
     - 功能按钮：导入/导出/新增按钮按规则2实现（文件大小/格式限制、进度条、二次确认弹窗等）。
     - 列表：包含序号列、最近一次操作人/时间、空值显示`--`、排序、分页、操作列按钮布局等。
     - 表单元素：文本框自动去空格、下拉框模糊搜索/清空、金额保留两位小数、手机号/邮箱格式校验、保存/提交按钮锁定/进度条/校验提示等。
     - 操作日志：在详情页或列表页底部展示操作日志表格，格式符合规则4-操作日志。
     - 数据脱敏：姓名、证件号码、手机号等敏感字段按规则5脱敏显示；同时支持特定角色查看明文（通过额外按钮/接口，需记录日志）。
   - 在HTML注释中注明交互逻辑、关联的功能ID以及遵循的规则条款（如`<!-- 文本框：规则4-文本框 -->`）。
4. 将所有界面整合到一个HTML文件中，通过导航栏或卡片式切换进行演示。
5. 使用无头浏览器（或模拟）生成每个界面的缩略图。
6. 返回HTML内容、缩略图和界面列表。

## 注意事项
- 生成的HTML应自包含，无需外部网络资源（除字体库外，可引用CDN）。
- 微交互需考虑用户体验，不宜过度。
- 代码应整洁，便于开发人员阅读参考。
- 必须严格遵循通用规则，审核环节将对此进行检查。
