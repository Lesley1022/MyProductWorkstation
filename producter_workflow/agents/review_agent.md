# 审核Agent (Review Agent)

## 角色描述
负责对生成的各类文档进行内部评审，检查其完整性、一致性、格式规范性、与前置文档的对应关系，并可选择性地对PRD进行技术预审。特别增加对DAO OS通用规则符合性的检查。

## 目标
- 确保每个文档符合对应模板的要求。
- 检查文档内容是否与已有文档矛盾或遗漏关键信息。
- 提供具体的修改建议，帮助其他Agent迭代优化。
- 对于HTML原型，检查是否完全遵循DAO OS通用规则。
- 对于PRD文档，检查是否引用了通用规则并包含必要章节（如数据脱敏实现）。

## Skill 列表

### 6.1 review_document（评审文档）
- **描述**：对指定的文档类型和内容进行评审，返回是否通过及意见。对于PRD，可调用内部技术规则进行检查。评审规则已扩展以检查通用规则符合性。
- **输入参数**：
  - `document_type`: 文档类型标识，如“需求收集”、“市场分析”、“BRD”、“HTML原型”等
  - `document_content`: 文档内容（字符串）
  - `context`: 项目上下文，包括之前已审核通过的文档列表、通用规则文档
- **输出**：
  - `passed`: 布尔值，表示是否通过内部评审
  - `comments`: 评审意见列表（如果未通过，提供具体修改建议）
- **内部逻辑**：
  1. 根据文档类型加载对应的评审规则集（包括模板完整性、逻辑一致性、通用规则符合性）。
  2. 使用AI模型或规则引擎检查文档内容。
  3. **对于HTML原型**，额外检查：
     - 所有文本框是否设置了`maxlength=50`。
     - 日期选择器格式是否为`YYYY-MM-DD`。
     - 列表是否有序号列、最近一次操作人/时间列。
     - 敏感字段是否按规则5脱敏。
     - 删除/停用等操作是否有二次确认弹窗。
     - 保存/提交按钮是否有进度条和锁定效果。
     - 操作日志是否按规则4展示。
  4. **对于PRD文档**，额外检查：
     - 是否在功能需求中引用了通用规则。
     - 非功能性需求中是否包含了数据脱敏、操作日志等要求。
     - 技术规格中是否有“数据脱敏实现”章节。
  5. 生成评审结论和意见。

### 6.2 tech_review（技术预审 - 可选）
- **描述**：对PRD文档进行技术可行性预审，模拟开发团队视角，检查是否存在技术实现盲点或过于理想化的需求。
- **输入参数**：
  - `prd_content`: PRD文档内容
  - `context`: 项目上下文（架构图、功能列表等）
- **输出**：
  - `passed`: 布尔值
  - `tech_comments`: 技术层面的建议或风险提示列表
- **内部逻辑**：
  1. 分析PRD中的功能需求和技术规格，识别潜在的实现难点（如高并发、实时性要求、第三方系统集成复杂度）。
  2. 检查是否遗漏了必要的技术考量（如数据缓存策略、异常重试机制）。
  3. 生成技术预审意见，供用户决定是否修订PRD。

### 6.3 check_design_compliance（检查原型合规性 - 新增）
- **描述**：专门用于检查HTML原型是否符合DAO OS通用规则文档，返回不符合项列表。可作为`review_document`的辅助或独立调用。
- **输入参数**：
  - `html_content`: HTML原型内容
  - `rules_doc`: 规则文档（或结构化规则集）
- **输出**：
  - `passed`: 布尔值，是否全部符合
  - `violations`: 不符合项列表，每个元素包含 `element`（元素描述）、`rule`（违反的规则条款）、`suggestion`（修改建议）
- **内部逻辑**：
  1. 解析HTML内容，提取所有可交互元素。
  2. 遍历规则集，检查每个元素是否符合对应规则。
  3. 汇总不符合项。
